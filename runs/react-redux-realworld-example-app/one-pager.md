# 클래스 컴포넌트에서 훅으로 — React Redux RealWorld 앱 마이그레이션

## 한 줄 요약

2018년경 스택에 머물러 있던 Create React App 기반 레거시 Medium 클론 프론트엔드(React 16 클래스 컴포넌트, Redux 3)를 하루 만에 Vite + React 17 함수형 컴포넌트로 완전히 전환했으며, 157개의 자동 테스트가 구버전과 신버전 모두에서 통과하여 동작 보존을 입증했습니다.

## 프로젝트 개요

### 무엇을 했는가

RealWorld 표준 스펙의 정식 "Conduit" 프론트엔드 구현(gothinkster/react-redux-realworld-example-app)을 마이그레이션했습니다. 이것은 다음을 포함하는 완전한 애플리케이션입니다:

- 사용자 인증(로그인, 회원가입, 로그아웃)
- 기사 작성, 수정, 삭제
- 댓글 추가/삭제
- 사용자 프로필 보기
- 즐겨찾기 기능
- 페이지 네이션

원본은 다음 스택으로 구성되어 있습니다:
- Create React App 1(Webpack 3/Babel 6)
- React 16.3 클래스 컴포넌트 13개
- Redux 3 + react-redux 5 (connect() 패턴)
- react-router-dom 4 + react-router-redux

### 왜 중요한가

이 앱은 (1) 정식 아카이브된 레거시 코드이므로 실제 도전 과제를 포함하고, (2) MIT 라이선스로 공개 포트폴리오 자산으로 활용 가능하며, (3) 원본에 테스트가 전혀 없어서(0개) 마이그레이션 전에 자동 테스트 스위트를 먼저 구축해야 하므로, 진정한 레거시 전환 도전을 보여줍니다.

## 진행 방식

### 1단계: 기존 동작 고정 (기초 다지기)

**마이그레이션 시작 전**에 원본 코드를 대상으로 하는 157개의 자동 테스트 스위트를 새로 작성했습니다.

- 리듀서 동작 테스트: 모든 액션 분기, 에러 경로, Redux 미들웨어 순서
- API 호출 테스트: URL 생성, 토큰 헤더, 응답 처리
- 사용자 흐름 테스트: 로그인부터 기사 작성, 댓글, 즐겨찾기까지 전체 시나리오

이 157개 테스트는 모두 원본 코드에서 통과했습니다(157/157 ✓).

### 2단계: 단계별 전환 (M0~M7)

원본에는 손도 대지 않으면서, 마이그레이션 작업을 7단계로 나누어 진행했습니다:

1. **M0 (도구 체인 변경)**: Vite로 전환, 새 버전의 React/Redux/라우터 설정
2. **M1-M2 (간단한 컴포넌트)**: 클래스 → 함수형 컴포넌트 변환
3. **M3 (폼 컴포넌트)**: useState 훅으로 상태 관리
4. **M4 (복잡한 생명 주기)**: Editor, Settings 컴포넌트의 useEffect 적용
5. **M5 (페이지 컴포넌트)**: Home, Article, Profile 전환
6. **M6 (앱 쉘)**: 인증, 리다이렉트 로직 훅화
7. **M7 (정리)**: 클래스 컴포넌트, connect() 사용 제거 확인

**매 단계마다** 157개 테스트를 실행하여 모두 통과하는지 확인했습니다. 한 단계라도 실패하면 그 단계를 다시 검토합니다.

### 3단계: 마이그레이션 후 검증

마이그레이션 완료 후, 동일한 157개 테스트를 신버전 코드에 실행했습니다 → **157/157 통과**.

이것이 핵심입니다: 같은 테스트가 구버전과 신버전 모두에서 통과한다는 것은, 두 버전이 사용자 입장에서 **정확히 똑같이 동작한다**는 입증입니다.

## 소요 시간 및 방식

- **소요 시간**: 하루 (2026-07-07)
- **방식**: AI 에이전트 파이프라인 (분석 → 테스트 생성 → 마이그레이션 → 리뷰 → 문서화)
- **총 코드 라인**: 약 2,340줄 (38개 모듈)

## 왜 이 방식이 안전한가?

### 전통적인 "코드 읽고 수정" 방식의 문제

손으로 코드를 읽고 아, 여기 클래스를 함수로 바꿔야겠다고 하면서 직접 수정하는 방식은:
- 엣지 케이스를 놓치기 쉬움 (비동기 처리, 미들웨어 상호작용 등)
- 뭔가 이상한지 알아채기 어려움 (테스트 없으면 특히)
- 같은 수정을 여러 곳에서 했을 때 일관성 유지가 힘듦

### 이 프로젝트의 안전 장치

1. **기존 동작을 먼저 테스트로 고정**: 마이그레이션 전에 "원본이 지금 이렇게 동작한다"는 것을 157개 테스트로 기록
2. **단계별로 변환**: 한 번에 하나의 모듈만 수정하고 테스트 실행
3. **자동 검증**: 매 단계마다 테스트가 모두 통과하는지 확인
4. **동작 보존 증명**: 마지막에 "같은 테스트가 구버전과 신버전 모두 통과"했으므로, 사용자가 느끼는 동작은 완전히 같음

이 방식은 마이그레이션 위험을 크게 낮춥니다. 빌드 도구와 의존성이 최신화되었지만, 사용자가 체감하는 동작은 그대로입니다.
